## Compact Rules

### Project Stack
- Idioma de interacción: responder siempre en español.
- Lenguaje: C++20 configurado en `CMakeLists.txt`.
- Plataforma: Windows 11 x64 only.
- Build: CMake + vcpkg + MSVC 2022.
- Shell preferida: PowerShell; no asumir Bash ni utilidades GNU.
- UI/runtime: aplicación desktop wxWidgets más el binario CLI `wmulecmd`.
- Networking/domain: eD2K + Kademlia + UPnP opcional.
- Testing: ejecutables MuleUnit registrados en CTest.

### Architecture & Structure
- Tratar el código como un **monolito modular en transición**, no como una hexagonal/clean ya implementada.
- `docs/blueprint.md` es el objetivo arquitectónico; `docs/PLAN_MODERNIZACION_2.0.md` es la hoja de ruta real.
- El entrypoint principal es `wmule.exe`; `wmulecmd.exe` usa External Connect y `amuleweb` es un adaptador opcional.
- El core operativo sigue concentrado en `CamuleApp` (`src/amule.cpp`) y depende de globals como `theApp` y `thePrefs`.
- Carpetas clave a respetar: `src/`, `src/libs/common/`, `src/libs/ec/`, `src/kademlia/`, `unittests/tests/`, `docs/`, `cmake/`.
- Seguir los límites actuales de CMake/librerías antes de inventar módulos top-level nuevos.
- Pedir cambios en términos de seams concretos, extracción incremental y reducción de acoplamiento.

### Code Conventions
- Mantener el bloque GPL en cada `.cpp` / `.h`.
- Usar include guards, no `#pragma once`, salvo convención local preexistente.
- Orden de includes: header correspondiente, headers del proyecto, headers de librería, standard library.
- Naming: clases `C*`, miembros `m_*`, funciones `CamelCase`, locales `camelCase`.
- Preferir C++ moderno: `true/false`, `nullptr`, `using`, `static_assert`, RAII, STL containers.
- Usar tabs para indentación y líneas de ~100 caracteres.
- Usar `wxT("...")` donde el código wxWidgets circundante lo espere.
- No expandir dependencias de wxWidgets a código que pueda quedar reusable/testeable.
- Favorecer helpers puros y validables como `Path`, `UPnPUrlUtils` y guards Kad/UDP.

### Safety Rules
- No editar archivos generados: `build/config.h`, `build/version.rc`, `src/libs/ec/cpp/ECCodes.h`, `src/libs/ec/cpp/ECTagTypes.h`, `pixmaps/flags_xpm/CountryFlags.h`.
- No committear artifacts de build, archivos de IDE ni salidas generadas.
- Mantener intactas las asunciones Windows/vcpkg/CMake y el build out-of-source.
- Priorizar siempre: **seguridad > estabilidad > arquitectura > nuevas features**.
- Mantener compatibilidad con protocolos existentes (eD2K, Kad, EC) salvo cambio planificado explícitamente.
- No presentar `docs/blueprint.md` como descripción fiel del presente.
- Evitar refactors arquitectónicos masivos mezclados con fixes funcionales.

### Testing & Validation
- Los tests viven en `unittests/tests/` y deben registrarse en `unittests/tests/CMakeLists.txt`.
- Priorizar cobertura MuleUnit focalizada para parsing, paths, EC, UPnP y concurrencia.
- CTest es el runner canónico; cada test también puede ejecutarse como binario standalone.
- `build-ninja/compile_commands.json` es la fuente de verdad para clangd.
- Mientras `docs/PLAN_MODERNIZACION_2.0.md` tenga la **Fase 2** abierta, priorizar:
  - normalización y validación de rutas externas
  - endurecimiento de External Connect y credenciales PBKDF2
  - feedback real de UPnP en UI/logs
  - tests de traversal, EC y UPnP
- Al cerrar cambios relevantes: build completo + `ctest` + verificación básica de `wmule.exe` y `wmulecmd.exe` + actualización documental si cambia el estado del plan.

### Git Workflow
- Trabajar en ramas feature, no directamente sobre `main`.
- Separar trabajo de seguridad riesgoso de refactors cosméticos cuando sea posible.
- Antes de PRs, la convención del proyecto pide `cmake --build . --config Debug` y `ctest --output-on-failure -C Debug` desde el directorio de build.
- Si un cambio mueve el estado de una fase, actualizar `docs/PLAN_MODERNIZACION_2.0.md` con validaciones y notas.

### Prompting Guidance
- Pedir cambios así: “extraé esta lógica de `CamuleApp` a un servicio testeable”, “encapsulá esta infraestructura detrás de una API local”, “reducí dependencias a `thePrefs`”, “introducí un seam para testear sin GUI/socket real”.
- Evitar prompts como: “agregalo al core hexagonal”, “implementalo en la capa de aplicación”, “conectá este adapter al domain kernel”, salvo que esas capas existan explícitamente en el cambio actual.
