## Internacionalización (i18n / NLS)

### Objetivo

Mantener todas las cadenas visibles al usuario dentro del sistema gettext existente basado en ficheros `.po/.mo`, evitando literales no traducibles en el código y asegurando que cualquier texto nuevo quede integrado en el flujo de traducción del proyecto.

### Decisión de arquitectura

- El proyecto mantiene como sistema oficial de internacionalización **gettext + catálogos `.po/.mo`**.
- No se debe introducir un sistema alternativo de i18n basado en JSON, INI, YAML, recursos embebidos u otro formato sin una decisión explícita de arquitectura.
- La gestión debe simplificarse mediante automatización del flujo, **no sustituyendo** el backend actual salvo decisión expresa del usuario.

### Reglas obligatorias para el código

- Toda cadena visible al usuario debe ser traducible.
- No introducir literales visibles al usuario “sin marcar” en código C++, wxWidgets, CLI, EC o webserver.
- Usar exclusivamente los patrones reconocidos por el sistema de extracción actual:
  - `_("<texto>")`
  - `wxTRANSLATE("<texto>")`
  - `wxPLURAL("<singular>", "<plural>", count)` o el equivalente ya establecido en el proyecto
- Usar `_()` para mensajes que deban resolverse directamente en runtime.
- Usar `wxTRANSLATE()` solo para textos estáticos o diferidos, por ejemplo títulos de columnas, arrays de labels o catálogos que luego se traduzcan explícitamente.
- No concatenar fragmentos traducibles si puede evitarse.
- Preferir cadenas completas con placeholders o formato parametrizado.
- Mantener consistencia terminológica con las traducciones ya existentes.
- Si se añade, elimina o modifica una cadena visible al usuario, el flujo i18n debe actualizarse en la misma tarea.

### Prohibiciones

- No añadir llamadas como `wxMessageBox("texto")`, `SetLabel("texto")`, `Append("texto")`, `AddPage(..., "texto")` o equivalentes con literales visibles al usuario sin marcar para traducción.
- No modificar manualmente los ficheros `.mo`.
- No dar por cerrada una tarea que introduce textos nuevos si los catálogos no han sido regenerados o sincronizados.
- No asumir que compilar el proyecto actualiza automáticamente los catálogos.
- No dejar cambios de UI o mensajes con `.po` desactualizados respecto al código.

### Flujo obligatorio cuando se añadan o modifiquen textos visibles

1. Añadir o modificar la cadena en el código usando `_()`, `wxTRANSLATE()` o plural.
2. Verificar que el fichero fuente afectado está incluido en `po/POTFILES.in`.
3. Regenerar la plantilla POT del proyecto.
4. Fusionar la POT actualizada dentro de todos los `.po`.
5. Compilar todos los `.po` a `.mo`.
6. Escribir o sincronizar los `.mo` en `assets/locale/<lang>/LC_MESSAGES/amule.mo`.
7. Copiar, si procede, los catálogos al árbol de build.
8. Verificar que el build afectado usa `ENABLE_NLS=ON`.
9. Validar visualmente o mediante comprobación del catálogo que la nueva cadena existe y es cargable por el runtime.
10. Actualizar documentación o notas de mantenimiento si el cambio altera el flujo de traducción.

### Scripts operativos

- `scripts/generate-pot.py` regenera `po/amule.pot` desde el código fuente usando `xgettext`.
- `scripts/compile-translations.py` compila cada `.po` a `.mo` y los sincroniza en `assets/locale/<lang>/LC_MESSAGES/`.
- Si se pide, `compile-translations.py` también puede copiar los catálogos al árbol de build.
- El flujo de traducción sigue siendo gettext puro: código → POT → PO → MO.

#### Objetivo del flujo

Automatizar de extremo a extremo el mantenimiento de i18n para evitar pasos manuales olvidados y garantizar que toda cadena nueva quede propagada desde el código hasta los catálogos ejecutables.

#### Requisitos del flujo

- Ejecutarse con Python.
- Usar rutas compatibles con Windows.
- Fallar con código de salida distinto de cero ante cualquier error.
- Mostrar mensajes claros y accionables.
- No modificar manualmente binarios fuera del flujo previsto.
- Poder usarse tanto manualmente como desde agentes o automatizaciones.

#### El flujo actual realiza como mínimo

1. Verificar que existen `po/` y `po2mo.py`.
2. Regenerar el `.pot` con `xgettext` cuando se use `generate-pot.py`.
3. Compilar cada `.po` en `assets/locale/<lang>/LC_MESSAGES/wmule.mo` y `amule.mo`.
4. Mantener compatibilidad con el dominio legado `amule.mo`.
5. Opcionalmente copiar los catálogos al árbol de build.
6. Validar que los ficheros generados existen realmente.
7. Emitir un resumen final con idiomas y dominios afectados.

#### Opciones recomendadas del compilador

- `scripts/generate-pot.py` no necesita opciones para el caso normal.
- `scripts/compile-translations.py --copy-to-build`
- `scripts/compile-translations.py --project-root K:\wMule --build-dir K:\wMule\build --copy-to-build`

#### Flujo de uso recomendado

1. Regenerar el POT:

   ```powershell
   python scripts\generate-pot.py
   ```

2. Compilar y copiar los catálogos:

   ```powershell
   python scripts\compile-translations.py --copy-to-build
   ```

### Reglas de uso para agentes y automatizaciones

- Todo cambio que introduzca o modifique cadenas visibles al usuario debe pasar por el flujo de traducciones disponible.
- No se considerará cerrada una tarea de UI, mensajes o etiquetas si los catálogos no han sido regenerados o sincronizados cuando corresponda.
- Si faltan herramientas o entradas en `POTFILES.in`, el agente debe informar del problema de forma explícita y no dejar el trabajo como completo.
- Si una cadena nueva no puede integrarse en gettext en esa misma tarea, debe indicarse como incidencia bloqueante.
- Los agentes deben preferir la automatización del flujo completo frente a pasos parciales manuales.

### Validación mínima obligatoria

- Build con `ENABLE_NLS=ON`
- Regeneración sin errores de POT, PO y MO
- Presencia de catálogos generados en:
  - `assets/locale/<lang>/LC_MESSAGES/wmule.mo`
  - `assets/locale/<lang>/LC_MESSAGES/amule.mo`
  - y, si aplica, en el árbol de build correspondiente
- Verificación básica del componente afectado
- Confirmación de que las nuevas cadenas no han quedado fuera del sistema de traducción

### Regla operativa resumida

Si el texto lo va a ver el usuario, debe entrar en gettext.

No debe introducirse ninguna cadena visible al usuario fuera del pipeline oficial de internacionalización del proyecto.
